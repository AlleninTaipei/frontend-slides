# LEARN.md

寫給要接手或想搞懂這個專案的人. 用平實的語言, 講清楚這個專案在做什麼, 為什麼要這樣做, 以及過程中踩過哪些坑.

## 一句話說明這個專案

`frontend-slides` 不是一個網站, 也不是一個 App. 它是一份寫給 AI coding agent 看的說明書 (`SKILL.md`), 教 agent 如何幫使用者做出一份漂亮的 HTML 投影片, 而且完全不需要 npm, 不需要框架, 一個 HTML 檔案就能在瀏覽器打開.

換句話說, 這個 repo 裡幾乎沒有會被"執行"的程式, 只有兩種東西:

1. 給 AI 讀的指令文件 (`SKILL.md` 及一堆支援文件).
2. 給人類使用者跑的小工具 (`scripts/` 底下的三支腳本, 負責轉檔, 部署, 匯出 PDF).

真正的投影片 HTML, 是使用者跟 agent 對話之後, 由 agent 當場生成的, 不會被存進這個 repo.

## 為什麼要做成這樣 (背後的哲學)

專案 README 裡寫了四條信念, 這四條其實可以當成整份文件的價值觀骨架:

1. 你不需要是設計師, 才能做出好看的東西. 你只需要"看到"然後反應.
2. 依賴是債務. 一個單純的 HTML 檔案十年後還能打開; 一個 2019 年的 React 專案, 祝你好運.
3. 通用即是平庸. 每份投影片都該像是為這個場合量身訂做的, 而不是套模板套出來的.
4. 註解是一種善意. 程式碼應該對"未來的你", 或任何打開這份檔案的人, 自我解釋.

這四條分別對應到專案裡三個具體的技術決策:

1. Zero Dependencies, 對應"依賴是債務". 每份投影片都是單一 HTML 檔, CSS 與 JS 全部內嵌.
2. Show, Don't Tell 的風格探索, 對應"你不需要是設計師". 與其問使用者"你喜歡什麼字體", 不如直接生成三份不同風格的標題投影片給對方看, 讓對方用眼睛選.
3. Anti AI Slop 設計哲學, 對應"通用即是平庸". `STYLE_PRESETS.md` 裡明文列出"不准用"的清單: Inter, Roboto, Arial 這類爛大街字體, 白底紫色漸層這種一眼就知道是 AI 生成的配色.

## 整體架構: 漸進式揭露 (Progressive Disclosure)

如果你只記得這個專案的一個技術概念, 記這個就好.

`SKILL.md` 本身很短, 只有一份"工作流程地圖", 分成 Phase 0 到 Phase 6 (偵測需求, 內容蒐集, 風格探索, 生成投影片, PPT 轉檔, 交付, 分享匯出). 它不會把所有細節都塞進自己身體裡, 而是像一份導覽手冊, 告訴 agent "如果你現在在 Phase 2, 去讀 `STYLE_PRESETS.md`; 如果你在 Phase 3, 去讀 `viewport-base.css` 跟 `html-template.md`".

這個設計不是憑空想像的, 而是踩過痛之後才長出來的. 看一下 git 歷史:

```
b206c50 Restructure skill for progressive disclosure — reduce context bloat ~70%
```

這次重構把 `SKILL.md` 從 1098 行砍到 183 行, 拆出 `STYLE_PRESETS.md`, `animation-patterns.md`, `html-template.md`, `viewport-base.css`. 原因很直白: AI agent 的"腦容量"(context window)是有限且昂貴的資源. 如果每次啟動這個技能, agent 都要把 1098 行的規則整份讀進去, 那大部分內容其實跟眼前這個任務無關, 純粹是浪費. 這跟人類團隊的知識管理其實是同一個道理, 想像一份新人手冊, 与其把整個公司的規章制度一次塞給新人, 不如先給一張地圖, 需要用到某個規定時再去查那一頁.

`bold-template-pack` 這個資料夾把這個原則做得更徹底, 分成三層:

1. `selection-index.json`, 極輕量的中繼資料 (34 個風格各自的 mood, tone, formality, best_for), agent 一開始只讀這份, 用來"篩選候選人".
2. `templates/<slug>/preview.md`, 每個風格一份輕量卡片, 只有在candidate 進入候選名單後才讀, 用來生成標題投影片預覽.
3. `templates/<slug>/design.md`, 完整的設計系統規格 (顏色, 字體, 間距, 元件文法), 只有在使用者"真的選定"這個風格之後, agent 才會讀進去, 拿來生成整份投影片.

也就是說, agent 永遠不會一次讀完 34 份完整設計文件, 這是刻意的節流設計, 而不是偷懶.

## 檔案地圖

專案根目錄大致長這樣, 每個檔案在整個流程裡各司其職:

```
SKILL.md                核心工作流程, agent 一定會先讀這份
STYLE_PRESETS.md         12 個安全預設風格 (顏色, 字體, 簽名元素)
viewport-base.css        強制性 CSS, 每份投影片都要完整貼進去
html-template.md         HTML 骨架與 JS 功能規格 (翻頁, 縮放, 行內編輯)
animation-patterns.md    動畫參考手冊, 依"情緒"對照到具體 CSS/JS 寫法
bold-template-pack/      34 種進階設計系統, 三層漸進式揭露
scripts/extract-pptx.py  把 .pptx 檔案的文字, 圖片, 備忘稿抽出成 JSON
scripts/deploy.sh        一鍵部署到 Vercel, 給使用者分享用
scripts/export-pdf.sh    用 Playwright 截圖每張投影片, 合併成 PDF
plugins/frontend-slides/ Claude Code 外掛封裝, 內容是根目錄檔案的鏡像複本
.claude-plugin/          Claude Code marketplace 的清單檔
```

值得注意的一點: `plugins/frontend-slides/skills/frontend-slides/` 底下的檔案, 跟根目錄的 `SKILL.md`, `STYLE_PRESETS.md` 等等內容是完全一致的 (可以用 `diff` 驗證). 這不是失誤留下的重複檔案, 而是刻意的雙軌設計, 根目錄的檔案是給"任何 coding agent"(包含 Codex, Gemini CLI 等等)手動安裝或直接讀取用的, `plugins/` 底下的複本則是專門封裝給 Claude Code 的 marketplace 外掛安裝流程用. 兩邊要保持同步, 修改其中一邊時記得也要同步另一邊.

## 固定 16 比 9 舞台模型 (Fixed Stage)

這是整個系統最硬性的規則, `SKILL.md` 裡明文寫"NON NEGOTIABLE". 所有投影片都用 1920 x 1080 的座標系統寫死, 不管使用者用手機還是筆電打開, 都是同一份 HTML, 靠 `transform: scale()` 把整個舞台整體縮放去塞滿視窗, 不夠的地方用黑邊補齊 (letterbox / pillarbox), 而不是像一般網頁那樣用 RWD 斷點去"重排"內容.

為什麼不做成響應式網頁常見的做法.因為投影片是一種"設計過的畫面", 每個字, 每張圖的相對位置都是精心安排的, 如果讓瀏覽器依照螢幕寬度重新排版, 排版師精心設計的視覺平衡就會被打散, 在手機上可能變成一團亂. 固定舞台模型犧牲了"手機上也能填滿螢幕"這件事, 換來"任何裝置看到的畫面永遠跟設計稿一致"這個更重要的保證.

`viewport-base.css` 就是這個模型的骨架, 裡面定義了 `.deck-viewport`(填滿瀏覽器視窗), `.deck-stage`(固定 1920x1080 的畫布, 由 JS 動態設定 transform), 以及 `.slide`(疊放在舞台裡, 用 `visibility` / `opacity` / `pointer-events` 控制顯示與隱藏). 這份檔案被明文要求"整份複製貼上"到每一份產出的投影片裡, 不能只抄一部分.

## 開發過程中踩過的坑

這幾個坑都留在 git log 裡, 每一個都對應到後來寫進規則文件裡的一條"鐵律". 讀懂這些坑, 比讀懂功能本身更有價值, 因為它們代表"別人已經替你撞過的牆".

### 坑一: 用 display none 切換投影片, 結果全部一起出現

```
7f32a26 Fix slide visibility switching
```

一開始很直覺的做法是用 `display: none` 隱藏不是目前的投影片, `display: block` 顯示目前的投影片. 問題是, 生成出來的投影片內容通常會有自己的版面樣式, 例如 `.slide-content { display: flex; }`, 這種"後寫的"CSS 規則, 在 CSS 特異性 (specificity) 上會蓋掉前面切換用的 `display: none`, 結果就是每張投影片同時疊在畫面上, 變成一團亂碼.

修法是改用 `visibility` + `opacity` + `pointer-events` 三個屬性的組合來控制顯示與隱藏, 而不是 `display`. 這三個屬性不會被"版面用的 display 規則"意外覆蓋, 因為它們處理的是完全不同的層面 (可見性與互動性, 而不是排版盒模型). 這條教訓後來被寫進 `SKILL.md` 變成一條硬性規則.

給後來者的啟示: 當你要用 CSS 做"切換顯示/隱藏"這種語意時, 優先選擇不會被排版樣式碰到的屬性 (`visibility`, `opacity`, `pointer-events`), 不要用 `display`, 因為 `display` 太容易被其他規則覆寫.

### 坑二: CSS 函式加負號, 瀏覽器不報錯但直接失效

```
0c231fd Add CSS function negation gotcha to prevent silent positioning bugs (#2)
```

寫 `right: -clamp(28px, 3.5vw, 44px);` 這種寫法, 想要把 `clamp()` 算出來的值取負號, 結果瀏覽器不會報錯, 但整條 CSS 宣告會被"靜默忽略", 元素的位置就悄悄跑掉, 而且開發者工具上完全看不出原因, 因為根本沒有錯誤訊息.

正確寫法是包一層 `calc(-1 * clamp(28px, 3.5vw, 44px))`. CSS 規範不允許在函式名稱前面直接加負號, 但透過 `calc()` 做乘法運算就沒問題.

給後來者的啟示: 這種"沒有報錯但行為不對"的坑最危險, 因為沒有任何工具會主動提醒你. 遇到元素位置怪異卻查不到錯誤時, 第一步該檢查的就是: 是不是哪裡直接對 CSS 函式取了負號.

### 坑三: SKILL.md 越寫越肥, 拖垮每次啟動的成本

```
b206c50 Restructure skill for progressive disclosure — reduce context bloat ~70%
```

前面"漸進式揭露"那節提過, 這裡補充一個更根本的道理: 文件會像任何活的東西一樣自然成長, 每次修一個小 bug 就順手加一段規則, 日積月累就從精簡的地圖膨脹成一本厚重的百科全書. 沒有人是故意把文件寫肥的, 但如果沒有人定期回頭"瘦身", 文件只會越長越大.

給後來者的啟示: 定期回頭檢查核心文件的長度跟結構, 是否還讀得完, 是否還聚焦. 這件事沒有工具可以自動幫你做, 得靠人 (或 agent) 主動意識到"這份文件已經太肥了", 然後動手拆分.

### 坑四: 重構時不小心把之前 PR 修好的功能弄丟了

```
384d1a0 Reintegrate features from PRs #19, #11, #2 lost during restructure
be267ee Reintegrate inline editing feature from PR #23
```

在做上面提到的"瘦身重構"時, 由於是把 1098 行的內容拆解重組, 過程中不小心漏掉了幾個先前 PR 已經修好的功能 (行內編輯, 圖片資料夾支援等等), 後來得靠額外兩次 commit 才補回來.

給後來者的啟示: 大規模重構, 尤其是"把一份大文件拆成好幾份小文件"這種結構性调整, 最容易在搬移過程中遺漏內容. 動手之前最好先列一份"重構前有哪些功能"的清單, 重構完再逐項核對, 而不是只憑印象搬移.

### 坑五: 匯出功能忘記清乾淨編輯狀態

```
ee6f18e Fix: clear navDots container and strip edit state in exportFile
```

投影片內建"行內編輯"功能(滑鼠移到左上角或按 E 鍵進入編輯模式), 使用者編輯完之後可以匯出檔案. 一開始的匯出邏輯忘記把"目前正在編輯中"這個暫時性的狀態, 以及畫面導覽用的 `navDots` 容器清乾淨, 結果匯出的檔案裡殘留了不該存在的執行期狀態.

給後來者的啟示: 任何"匯出/存檔"功能, 都要把它想成是在做一次乾淨的快照, 匯出前要主動歸零所有暫時性的 UI 狀態 (編輯模式開關, 動態產生的 DOM 容器等等), 不能只是把目前的 DOM 原封不動存下來.

## 值得學起來的優秀工程習慣

讀完這個專案的演進歷史, 有幾個做法值得內化成自己的習慣, 不只適用在這個專案:

1. 把限制寫成"規則", 而不是靠記憶. 每次踩到一個坑, 都立刻把教訓寫成 `SKILL.md` 或 `STYLE_PRESETS.md` 裡的一條明文規則 (例如 CSS 取負號那條, 就直接寫進"CSS Gotchas"章節). 這樣下一次不管是人還是 AI 接手, 都不需要重新撞一次牆.
2. 主動幫消費者(這裡的消費者是 AI agent)節省資源. 漸進式揭露的核心精神, 是意識到"讀者的注意力/記憶體是有限的", 因此主動設計資訊的揭露順序, 而不是把所有東西一次攤開.
3. 用清單而非期待"記得住". 三層式的 bold template pack (先讀 index, 再讀 preview, 最後才讀 design)是一種流程上的守門機制, 用檔案結構本身去"強迫"正確的閱讀順序, 而不是靠一句"請不要一次讀太多"的提醒去期待對方自律.
4. 重構前先盤點, 重構後要核對. PRs #19, #11, #2 遺失又補回的教訓說明了這件事, 光靠印象做大搬移是靠不住的.
5. 依賴是要付利息的債務. 這個專案選擇"零依賴, 單一 HTML 檔"這條看似比較土炮的路線, 換來的是十年後這份檔案大概率還能直接打開, 不需要煩惱 npm 套件版本相容性, 這是一種對"長期可維護性"的刻意取捨, 而不是技術能力不足的妥協.

## 給初學者的一個比喻

如果把這個 repo 想像成一間餐廳: `SKILL.md` 是外場服務生手上那張精簡的菜單, 顧客(使用者)一坐下, 服務生不會把整本食譜甩給你看, 而是先問你想吃什麼口味, 然後才去後場查對應的那份食譜(`STYLE_PRESETS.md` 或某個 `design.md`). 後場的每份食譜都寫得非常仔細, 包含配色, 字體, 動畫節奏, 這些都是廚師(agent)真正動手做菜時才需要的細節, 顧客不需要, 服務生平常也不需要背下來.

而 `viewport-base.css` 則像是這間餐廳"不管外送到哪裡, 餐盒的尺寸永遠一致"的規定, 不管顧客是在家裡用大餐桌吃, 還是在捷運上用手機打開, 端出來的那道菜(投影片)長相都應該一模一樣, 不會因為場地不同就隨便把擺盤打散重排.
